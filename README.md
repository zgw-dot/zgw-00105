# 售后退货异常分析看板

一个全栈数据应用，用于分析售后退货过程中的异常情况，包括超期退货、重复退货和质检冲突。

## 技术栈

- **前端**: React 18 + TypeScript + Vite + TailwindCSS 3
- **后端**: Express 4 + TypeScript
- **状态管理**: Zustand 4
- **路由**: React Router DOM 6
- **图表**: Recharts 2
- **CSV 解析**: Papa Parse 5
- **图标**: Lucide React

## 功能特性

### 核心流程
1. **CSV 导入**: 支持订单、退货、质检三份 CSV 表的导入，提供示例数据一键加载
2. **字段映射**: 智能中英文列名自动映射，支持手动调整和版本化保存
3. **规则配置**: 可视化配置超期天数、重复退货窗口、质检冲突类型
4. **异常分组**: 按超期、重复、冲突三类规则生成异常批次与明细
5. **运行历史**: 保存所有运行记录，支持历史查询和跨轮对比
6. **坏行隔离**: 自动识别并隔离坏数据，提供完整错误清单
7. **报告导出**: 支持 CSV/HTML/JSON 三种格式，导出元数据可追溯

### 失败路径处理
| 错误类型 | 处理方式 | 说明 |
|---------|---------|------|
| 缺少必需列 | 整批拒绝 | 无法进行后续分析，需修正数据后重新上传 |
| 日期格式错误 | 坏行隔离 | 隔离错误行，正常行继续分析 |
| 非法规则值 | 规则校验失败 | 不允许保存，提示修正参数 |
| 重复订单号 | 坏行隔离 | 隔离重复行，保留第一条记录 |

### 一致性链路
- 所有数据持久化到本地文件系统
- 应用重启后可查看：历史映射、规则版本、运行历史、坏行摘要、最近导出
- 导出文件包含完整元数据：runId、mappingId、rulesId，可精确追溯

1. 打开浏览器访问 http://localhost:5173
2. 点击左侧导航栏的「数据导入」
3. 点击「加载示例数据」按钮
4. 验证：
   - 三张表（订单、退货、质检）数据加载成功，每张表 20 行
   - 自动完成字段映射
   - 自动保存默认规则
   - 自动运行异常分析
   - 自动跳转到主看板

### 步骤 2: 查看主看板

1. 验证概览卡片：
   - 显示总异常数、超期数、重复数、冲突数
   - 显示与上一轮运行的对比变化（首次运行显示 0）
2. 验证异常分布：
   - 饼图展示三类异常占比
   - 柱状图展示各类异常数量
3. 验证分组列表：
   - 按异常类型分组显示
   - 每组显示数量和涉及订单
   - 支持展开/折叠分组

### 步骤 3: 查看异常详情

1. 点击任意分组中的「查看详情」
2. 验证详情抽屉：
   - **原始行**：显示原始 CSV 行数据
   - **质检记录**：关联的质检报告
   - **错误清单**：检测到的具体错误
   - **处理说明**：建议的处理方式
   - **隔离原因**：如果是坏行，显示隔离原因

### 步骤 4: 调整规则重新分析

1. 点击左侧导航栏的「规则配置」
2. 修改规则参数：
   - 超期天数：从 15 改为 10
   - 重复退货窗口：从 30 改为 60
   - 质检冲突类型：增加或减少类型
3. 点击「保存规则」
4. 点击「重新分析」
5. 验证：
   - 产生新的运行记录
   - 看板显示新的异常数量
   - 对比卡片显示与上一轮的数量变化
   - 运行历史中可看到两条记录

### 步骤 5: 查看运行历史

1. 点击左侧导航栏的「运行历史」
2. 验证：
   - 显示所有运行记录列表
   - 每条记录包含：运行ID、时间、状态、异常总数、文件信息
   - 支持加载历史运行结果
   - 支持对比两次运行结果

### 步骤 6: 查看坏行隔离

1. 在主看板或运行历史中查看坏行摘要
2. 验证：
   - 显示坏行总数和各类型错误数量
   - 坏行列表包含：行号、原始数据、错误类型、错误描述、隔离原因
   - 区分整批拒绝和部分隔离场景

### 步骤 7: 导出报告

1. 点击左侧导航栏的「报告导出」
2. 选择导出格式（CSV/HTML/JSON）
3. 选择要包含的内容（异常明细、坏行列表、规则配置、统计摘要）
4. 点击「创建导出」
5. 验证：
   - 导出任务完成后显示下载链接
   - 下载文件内容完整
   - 导出历史中可查看所有导出记录
   - 查看导出元数据，确认包含 runId、mappingId、rulesId

### 步骤 8: 验证一致性链路

1. 记录当前的：
   - 字段映射配置
   - 规则参数
   - 运行结果
   - 导出的文件
2. 刷新浏览器页面
3. 验证：
   - 字段映射配置仍然存在
   - 规则版本历史完整
   - 运行历史记录完整
   - 最近导出的文件可下载
4. 重启后端服务（停止后重新运行 `npm run dev:api`）

### 步骤 9: 测试失败路径

1. **缺少必需列测试**：
   - 创建一个缺少 `orderId` 列的 CSV 文件
   - 尝试导入
   - 验证：系统提示「缺少必需列：orderId」，整批拒绝

2. **日期格式错误测试**：
   - 修改示例数据中的某行日期为无效格式（如「abc」）
   - 导入并分析
   - 验证：该行被隔离为坏行，错误类型为「invalid_date」

3. **非法规则值测试**：
   - 在规则配置中设置超期天数为 -1
   - 验证：系统提示「超期天数必须大于 0」，不允许保存

4. **重复订单号测试**：
   - 创建包含重复 orderId 的 CSV 文件
   - 导入并分析
   - 验证：重复行被隔离，错误类型为「duplicate_order」

## API 接口

### 数据导入
- `GET /api/import/sample` - 获取示例数据
- `POST /api/import/save` - 保存上传的文件
- `POST /api/import/validate` - 验证文件结构

### 字段映射
- `GET /api/mapping` - 获取历史映射列表
- `GET /api/mapping/:id` - 获取指定映射
- `POST /api/mapping/auto` - 自动映射
- `POST /api/mapping` - 保存映射

### 规则配置
- `GET /api/rules` - 获取当前规则和历史
- `POST /api/rules` - 保存规则
- `POST /api/rules/validate` - 验证规则

### 异常分析
- `POST /api/analyze/run` - 执行分析
- `GET /api/analyze/history` - 获取运行历史
- `GET /api/analyze/:id` - 获取指定运行结果
- `GET /api/analyze/latest` - 获取最新结果
- `GET /api/analyze/compare/:id1/:id2` - 对比两次运行

### 报告导出
- `POST /api/export/create` - 创建导出
- `GET /api/export/history` - 获取导出历史
- `GET /api/export/download/:id` - 下载导出文件

## 数据存储结构

所有数据存储在 `data/` 目录下：

```
data/
├── uploads/                 # 上传的原始 CSV 文件
├── mappings/                # 字段映射版本
│   └── mapping_<id>.json
├── rules/                   # 规则版本
│   └── rules_<id>.json
├── runs/                    # 运行结果
│   └── run_<id>.json
├── badrows/                 # 坏行记录
│   └── badrows_<runId>.json
├── exports/                 # 导出文件
│   ├── export_<id>.csv
│   ├── export_<id>.html
│   └── export_<id>.json
└── indexes/                 # 索引文件
    ├── mappings.json
    ├── rules.json
    ├── runs.json
    └── exports.json
```

## 异常类型说明

| 类型 | 颜色 | 触发规则 |
|------|------|---------|
| 超期退货 | 橙色 #f59e0b | 退货申请日期 - 下单日期 > 超期天数 |
| 重复退货 | 蓝色 #3b82f6 | 同一订单号在重复退货窗口内多次申请退货 |
| 质检冲突 | 红色 #ef4444 | 退货原因与质检结果不匹配 |

## 常见问题

**Q: 数据存储在哪里？安全吗？**
A: 所有数据存储在本地 `data/` 目录，不会上传到任何服务器。

**Q: 如何重置所有数据？**
A: 删除 `data/` 目录即可，系统会自动重新创建。

**Q: 支持哪些日期格式？**
A: 支持 YYYY-MM-DD、YYYY/MM/DD、MM/DD/YYYY 等常见格式。

**Q: 可以自定义异常规则吗？**
A: 可以在「规则配置」页面调整参数，三种检测逻辑的阈值均可配置。
